## Recommendations

1. **Security first.** Review every `no-auth` external endpoint above. The Qdrant Cloud URLs and the `ml.dctrl.ai/api/sql/execute-query` SQL endpoint are called without credentials - confirm they are protected at the network layer, or add an API key. Move hardcoded hosts/IPs (Tailscale IPs, Qdrant URLs) into environment variables like the workflows that already use `{{#env.*#}}`.
2. **Cut the clutter.** Delete or archive the `LOW-VALUE` workflows (`гЂђsameгЂ‘*` duplicates and `*_test` scratch apps). This is the single biggest reduction in maintenance surface.
3. **Consolidate the RAG families.** The `vp_rag_*` and `DCTRL-Demo_*RAG*` families are many slight variants of one pipeline. Pick a canonical version per family and retire the rest.
4. **Add error handling** (retry and/or a fail-branch/default value) to the LLM, code, and HTTP nodes in the `NEEDS-WORK` workflows so a single upstream hiccup doesn't fail the whole run.
5. **Write a one-line description** for every kept workflow. All 53 are blank; this is what powers the tracker and makes the catalog usable.
6. **Fix orphan/placeholder nodes** flagged above (disconnected nodes, the `Lie about everything` test prompt in `test_workflow`).

## Methodology & limitations

- Source: the exported DSL (`*.yml`) only - no workflows were executed, so this cannot judge runtime correctness, output quality, or whether an endpoint is reachable.
- "No error handling" means no node has `retry_config.retry_enabled` or an `error_strategy` other than `none`; a workflow may still be fine if failures are acceptable.
- Secret detection is pattern-based (AWS/OpenAI/Anthropic keys, bearer tokens, JWTs, connection strings). It found none, but absence of a match is not a guarantee.
- Re-run anytime with `python src/analyze_workflows.py` after re-exporting.
